# Spotify Listening Stats

A private website for exploring your Spotify Extended Streaming History.
Browse **artists, albums, and songs** in sortable tables, see listening trends,
your daily listening clock, genre breakdowns, and any date range. Your listening
data is processed locally / in your own browser — nothing is uploaded anywhere.

Built from `Streaming_History_Audio_*.json` (the "Extended streaming history"
export you request from Spotify → Privacy settings).

**Live (hosted) version:** https://matthewmach.github.io/spotify_stats/ — loads
your files in your browser (stats only; no genres/art). Run it locally for the
full experience with cover art and genres.

There are two ways to run it:

- **Locally** with Python (full features, incl. optional genres + cover art).
- **Hosted on GitHub Pages** as a static site where each visitor loads their own
  files in the browser (stats only — no genres/art; see *Deploy* below).

## Quick start (local)

```
cd stats
python build.py     # aggregate the JSON history -> data/data.json
python serve.py     # opens the site in your browser
```

Or just double-click **`run.bat`** on Windows (builds if needed, then launches).

> **Always launch via `run.bat` / `serve.py` (an `http://localhost` URL).** If you
> open `index.html` directly (a `file://` URL), browsers disable `fetch` and
> IndexedDB, so the page can't auto-load `data.json` or remember files you pick —
> you'd get the upload screen every time.

The history folder is read from `build.py`'s `HISTORY_DIR`. It currently points to:

```
G:\Documents\Music\my_spotify_data\Spotify Extended Streaming History
```

Change that constant (or set the `SPOTIFY_HISTORY_DIR` env var) if your export
lives elsewhere.

## Optional: genres + cover art (Spotify API)

The streaming history has no genres or images — those come from the Spotify Web API.

1. Go to https://developer.spotify.com/dashboard → **Create app**.
   - Redirect URI can be anything, e.g. `http://localhost:8888` (unused).
2. Open the app → **Settings** → copy the **Client ID** and **Client Secret**.
3. Copy `config.example.json` → `config.json` and paste them in.
4. Run the enrichment (uses the no-login Client Credentials flow):

```
python enrich.py
```

This fetches genres + artist/album/track cover art, caches them in
`data/metadata.json`, and merges them into `data/data.json`. Re-running is
incremental (cached IDs are skipped). Refresh the browser to see images, the
**Genres** tab, and Spotify popularity scores.

## What you get

- **Overview** — hours, plays, streams, active days, listening-over-time chart,
  listening clock (by hour), by weekday, new-artist discovery, and top lists.
- **Artists / Albums / Songs** — sortable, searchable tables. Click any row for a
  detail panel (per-year plays, top tracks, skip/completion rates, genres).
- **Genres** — top genres by hours listened (requires enrichment).
- **Date filter** (filter bar) — scope the entire site to any range:
  - quick **year chips** (All time, 2020 … 2025),
  - a **From / To calendar** picker for any custom day range,
  - **Quick ▾** presets (last 30 / 90 days, last 6 / 12 months).
  Every table, chart, and stat recomputes live for the selected window.

## Local vs. hosted — the mode badge

The header shows which build you're looking at:

| Badge | Meaning |
|-------|---------|
| 🟢 **Local** | Served locally; loaded from `data/data.json` (genres + cover art available). |
| 🔵 **Pages** | The hosted `docs/` build; always loads files in your browser, no `data.json`. |
| 🟠 **Browser** | A local build where you loaded files by hand instead of `data.json`. |

## Deploy to GitHub Pages (static, no server)

`web/` is a dependency-free static site. When there's no `data/data.json` (the
case on Pages), it shows an **upload screen**: each visitor picks their own
`Streaming_History_Audio_*.json` files, which are parsed and aggregated
**entirely in their browser** (`build.py`'s logic is mirrored in JS) and cached
in **IndexedDB** so they don't re-pick on every visit.

A ready-to-publish copy lives in **`docs/`**, generated by:

```
python deploy.py     # copies web/ -> docs/ and injects the "pages" build flag
```

The only difference in `docs/` is a `<meta name="build" content="pages">` tag,
which forces the in-browser upload mode and the blue **Pages** badge.

Steps:

1. Make `stats/` a git repo (or push its contents), then run `python deploy.py`.
2. Commit `docs/` and enable **Settings → Pages → Deploy from branch →
   `main` / `/docs`**.
3. Visit `https://<you>.github.io/<repo>/` and load your files.
4. Re-run `python deploy.py` whenever you change anything in `web/`.

Notes:

- `.gitignore` excludes `config.json`, `data/data.json`, and
  `data/metadata.json`, so **your personal listening data and credentials are
  never published** — visitors only ever see their own uploaded files.
- The site reads files via `FileReader`; nothing is sent to any server (except
  the optional Spotify sign-in below, which talks directly to Spotify).

## Genres + cover art on the web (Spotify sign-in)

The hosted site can pull genres + art **without any client secret** using
Spotify's **PKCE** OAuth flow. After loading your files, click **"♪ Add genres &
art"**, sign in, and it fetches metadata directly from Spotify in your browser
(the Client ID is public — safe to ship; no secret is involved).

To enable it on your deployment, in the
[Spotify dashboard](https://developer.spotify.com/dashboard) → your app →
**Settings → Redirect URIs**, add the page URL(s):

- `https://<you>.github.io/<repo>/` (and `…/index.html`)
- for local testing: `http://127.0.0.1:8765/web/index.html`
  (Spotify forbids `localhost` — use the `127.0.0.1` form and open the site
  at that address)

> **Development-mode limit:** while your Spotify app is in "development mode",
> only the **app owner** and up to **25 allowlisted users** (added under
> Settings → User Management) can sign in. That's fine for personal use; a fully
> public sign-in needs Spotify to grant an extended-quota request.

Locally you can skip sign-in entirely and run `python enrich.py` instead.

## Files

| File | Purpose |
|------|---------|
| `build.py` | Aggregates the raw JSON into `data/data.json` (local workflow) |
| `enrich.py` | Pulls genres + art from the Spotify API (local only) |
| `serve.py` | Local static server + opens the browser |
| `deploy.py` | Generates the `docs/` GitHub Pages copy from `web/` |
| `run.bat` | One-click: build (if possible) + launch the server |
| `Install Desktop Shortcut.bat` | Puts a "Spotify Stats" shortcut on your Desktop |
| `web/` | The site — local source |
| `web/shared-build.js` | Browser port of `build.py` (chunked aggregation) |
| `web/spotify.js` | Optional in-browser enrichment via Spotify PKCE sign-in |
| `docs/` | Generated Pages build (commit this to publish) |
| `config.json` | Your Spotify credentials, for `enrich.py` (git-ignored) |

When you load files in the browser, parsing + aggregation run in a **Web
Worker** with a live progress screen; the stats only appear once it's finished.

The site loads its data in this order (local build): `data/data.json` →
browser-cached upload (IndexedDB) → otherwise the upload screen. The hosted
`docs/` build skips `data.json` and always uses the upload + cache.

## Metric notes

- A **stream** counts a play of 30s+ (the same threshold Spotify uses).
- **Skip %** / **Completion %** use the export's `skipped` flag and
  `reason_end == "trackdone"`.
- Hours in the listening clock are **UTC** (the export stores UTC timestamps).
